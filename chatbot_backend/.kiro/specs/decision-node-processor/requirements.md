# Requirements Document

## Introduction

The chatbot workflow engine needs to support a new `decision` node type generated by the frontend visual builder. A decision node acts as a conditional routing point in the workflow graph — it evaluates conditions on its outgoing transitions against session context and directs execution to the first matching target node. Unlike message or input nodes, it produces no user-visible output and does not pause execution. It simply determines the next path and lets the engine continue.

## Glossary

- **Decision_Node**: A workflow node with `type` equal to `"decision"`. It has no `config` and acts purely as a conditional routing point between an upstream node and multiple downstream targets.
- **DecisionNodeProcessor**: A Spring `@Component` that implements the `NodeProcessor` interface and handles Decision_Node processing.
- **Workflow_Engine**: The `WorkflowExecutionServiceImpl` service that orchestrates node processing, transition resolution, and session state management.
- **ConditionEvaluator**: An existing Spring component that evaluates simple expressions (e.g., `variable == value`) with `and`/`or` logical connectors against a context map.
- **Session_Context**: A `Map<String, Object>` stored on `ChatSession` that holds variables accumulated during workflow execution.
- **Transition**: A JSON object in the workflow definition containing `sourceNodeId`, `targetNodeId`, and `condition` fields that define edges between nodes.
- **Targeted_Routing**: The mechanism where the Workflow_Engine reads `_targetNodeId` from Session_Context to override default first-transition resolution.
- **WorkflowRepository**: A Spring Data JPA repository providing access to persisted workflow definitions including their JSONB workflow graph.
- **NodeProcessor**: The interface defining `canHandle(node)` and `process(node, session, placeholderService)` methods that all node processors implement.

## Requirements

### Requirement 1: Decision Node Identification

**User Story:** As a workflow engine, I want to identify decision nodes by their type field, so that the correct processor handles conditional routing.

#### Acceptance Criteria

1. WHEN a node with `type` equal to `"decision"` (case-sensitive) is evaluated, THE DecisionNodeProcessor SHALL return `true` from `canHandle`.
2. WHEN a node with `type` not equal to `"decision"` or with a null/missing `type` field is evaluated, THE DecisionNodeProcessor SHALL return `false` from `canHandle`.
3. THE DecisionNodeProcessor SHALL be registered with a Spring `@Order(5)` annotation value, placing it after WorkflowNodeProcessor (`@Order(4)`) and before any fallback processor in the evaluation chain.

### Requirement 2: Load Outgoing Transitions

**User Story:** As a workflow engine, I want the decision node processor to load outgoing transitions from the workflow definition, so that it can evaluate which path to take.

#### Acceptance Criteria

1. WHEN the DecisionNodeProcessor processes a Decision_Node, THE DecisionNodeProcessor SHALL load the Workflow entity from WorkflowRepository using the session's `workflowId` and retrieve its `workflowJson` field.
2. WHEN the DecisionNodeProcessor loads the workflow JSON, THE DecisionNodeProcessor SHALL extract all transitions from the `transitions` array where `sourceNodeId` equals the Decision_Node's `id`, preserving the order in which they appear in the JSON array.
3. IF the Workflow entity cannot be found in WorkflowRepository or its `workflowJson` field is null, THEN THE DecisionNodeProcessor SHALL send an error message indicating the workflow is no longer available via SimpMessagingTemplate to `/topic/chat/{sessionId}` and return a PAUSE action.
4. IF the Decision_Node has zero outgoing transitions after filtering, THEN THE DecisionNodeProcessor SHALL send an error message indicating the decision node has no outgoing transitions via SimpMessagingTemplate to `/topic/chat/{sessionId}` and return a PAUSE action.

### Requirement 3: Condition Evaluation with First-Match-Wins

**User Story:** As a workflow designer, I want conditions on decision node transitions evaluated in order so that the first matching path is taken, giving me predictable routing behavior.

#### Acceptance Criteria

1. WHEN the DecisionNodeProcessor has outgoing transitions, THE DecisionNodeProcessor SHALL evaluate each transition's `condition` field against Session_Context using ConditionEvaluator in the order the transitions appear in the workflow JSON (array index 0 first, then 1, 2, etc.).
2. WHEN a transition's `condition` field is null or empty, THE DecisionNodeProcessor SHALL skip that transition and proceed to evaluate the next transition in order.
3. WHEN a transition's condition evaluates to `true`, THE DecisionNodeProcessor SHALL store that transition's `targetNodeId` as `_targetNodeId` in Session_Context and stop evaluating remaining transitions (first-match-wins).
4. WHEN the DecisionNodeProcessor stores `_targetNodeId` in Session_Context, THE DecisionNodeProcessor SHALL return a `NodeProcessingResult` with action `CONTINUE` and a `null` response.
5. IF the matching transition's `targetNodeId` is null or empty, THEN THE DecisionNodeProcessor SHALL send an error message indicating the matched transition has no target node via SimpMessagingTemplate to the session's topic and return a `NodeProcessingResult` with action `PAUSE` and a `null` response.

### Requirement 4: No-Match Error Handling

**User Story:** As a workflow designer, I want clear error feedback when no transition condition matches, so that I can diagnose misconfigured decision nodes.

#### Acceptance Criteria

1. IF no transition condition evaluates to `true` for a Decision_Node, THEN THE DecisionNodeProcessor SHALL send a `ChatErrorResponse` with the message "No matching condition found for decision node" via SimpMessagingTemplate to `/topic/chat/{sessionId}`.
2. IF no transition condition evaluates to `true`, THEN THE DecisionNodeProcessor SHALL return a `NodeProcessingResult` with action `PAUSE` and a `null` response.
3. IF no transition condition evaluates to `true`, THEN THE DecisionNodeProcessor SHALL NOT store any `_targetNodeId` key in Session_Context.

### Requirement 5: Silent Pass-Through Behavior

**User Story:** As a chatbot user, I want decision nodes to route me without sending visible messages, so that the conversation feels natural and uninterrupted.

#### Acceptance Criteria

1. WHEN a transition's condition evaluates to `true`, THE DecisionNodeProcessor SHALL return a `NodeProcessingResult` with a `null` ChatResponse (no message sent to user).
2. WHEN a transition's condition evaluates to `true`, THE DecisionNodeProcessor SHALL NOT send any `ChatResponse` or `ChatErrorResponse` message to `/topic/chat/{sessionId}` via SimpMessagingTemplate.

### Requirement 6: Integration with Workflow Engine Routing

**User Story:** As a workflow engine, I want the decision node processor to use the existing targeted routing mechanism, so that no changes are required to the core engine loop.

#### Acceptance Criteria

1. WHEN the DecisionNodeProcessor stores `_targetNodeId` in Session_Context, THE Workflow_Engine SHALL use that value to resolve the next node instead of using the default first-transition lookup.
2. WHEN the Workflow_Engine consumes `_targetNodeId` from Session_Context, THE Workflow_Engine SHALL remove the `_targetNodeId` key from Session_Context immediately after resolution, before processing the resolved node.
3. THE DecisionNodeProcessor SHALL rely solely on the existing `_targetNodeId` mechanism for routing and SHALL NOT modify any other engine behavior or session state fields.

### Requirement 7: Component Registration and Ordering

**User Story:** As a developer, I want the decision node processor to follow the same component pattern as other processors, so that it integrates cleanly with the existing processor discovery mechanism.

#### Acceptance Criteria

1. THE DecisionNodeProcessor SHALL be annotated with `@Component` for Spring auto-detection.
2. THE DecisionNodeProcessor SHALL be annotated with `@Order(5)` to place it after existing processors (InputNodeProcessor at 1, MessageNodeProcessor at 2, ApiNodeProcessor at 3, WorkflowNodeProcessor at 4).
3. THE DecisionNodeProcessor SHALL use constructor injection for all dependencies (WorkflowRepository, ConditionEvaluator, SimpMessagingTemplate).
4. THE DecisionNodeProcessor SHALL implement the `NodeProcessor` interface, providing `canHandle(Map<String, Object> node)` and `process(Map<String, Object> node, ChatSession session, PlaceholderService placeholderService)` methods.
